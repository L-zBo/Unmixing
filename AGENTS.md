# 项目协作规范

## Git提交与任务记录规范

- 每次收到用户明确需求并完成改动后，**先准备 commit 但不立即执行**：展示`git status`/`git diff`与草拟的 commit message，由用户核对内容和 message 后再执行`git commit`。
- 每次`git commit`完成后，**`git push`前必须口头告知目标 remote/branch**，由用户确认后才执行`git push`；用户未明确要 push 则只 commit 不 push。
- 每次提交只允许包含本次需求直接相关的改动，不得混入无关文件、历史遗留改动或顺手修改。
- 代码改动与论文/文档改动必须分开提交，不得混在同一次提交中。
- 如果一次需求包含多个彼此独立的子任务，则每个独立任务完成后分别提交。
- 提交前必须检查`git status`和`git diff`，确认提交范围准确无误。
- 如果工作区中已有未提交改动，提交时只能纳入本次任务相关文件，不得把旧改动一并提交。
- 若`git status`显示 working tree clean，或本次需求只是阅读、分析、总结、答疑，没有产生实际文件改动，**严禁触发任何形式的`git commit`（含`--allow-empty`）**，必须在回复中明确说明”本次无文件改动，未提交”。
- 回顾项目进展时，必须同时查看`git`提交历史和当前未提交改动，不能只依据其中一方下结论。
- 对于用户明确提出的长期协作规则，在当前项目内默认持续生效，后续任务无需用户重复提醒。

## 提交信息要求

每次提交信息都必须清楚记录以下内容：

- 本次用户要求是什么
- 实际完成了哪些改动
- 涉及了哪些文件
- 如有未完成项、风险或说明，也要简要写明

## 推荐提交格式

标题：
`<本次任务的一句话总结>`

正文：
`用户要求：<用户本次指令摘要>`
`实际改动：<本次完成内容摘要>`
`涉及文件：<相对路径1>、<相对路径2>、<相对路径3>`
`说明：<可选，填写未完成项/风险/额外说明>`

## 执行原则

- **严禁空 commit**：`git status`显示 working tree clean 时不得以任何理由触发`git commit`（含`--allow-empty`）。
- **严禁以 user prompt 原文作为 commit message**：任何形如`[Claude Code] <prompt原文> prompt #N`或类似"抄 prompt"的格式一律禁止；commit message 必须由 AI 根据实际 diff 手写，反映真实改动。
- **commit 是协作动作，不是 turn 收尾**：不得把"一次 user turn 结束"当成 commit 触发点，只有实际完成需求且用户确认后才 commit。
- 不为了凑次数而乱提交。
- 不把一次命令执行当成一次提交。
- 不写无意义的流水账提交。
- 提交历史必须能清楚反映"用户要求了什么、实际改了什么"。
